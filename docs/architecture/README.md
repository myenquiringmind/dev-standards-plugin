# Architecture

This folder explains **how `dev-standards-plugin` is structured** and **why**. It is split into three sub-folders following the Diataxis taxonomy:

- **`principles/`** — explanation. Core concepts the framework is built on. Load-bearing for every agent working in this repo.
- **`components/`** — reference. Catalog of what actually exists: agents, commands, hooks, schemas, skills, rules, MCP servers. Populated as components materialize.
- **`lifecycle/`** — explanation. How work flows through the development lifecycle: discover → research → design → develop → test → validate → deploy → operate → maintain. Populated as phases materialize.

## Start with principles

The principles folder is the most important. Every agent working in this repo loads the relevant principle files via `@include` from CLAUDE.md. Without principles, the rest of the architecture is arbitrary.

**The ten load-bearing principles:**

1. **Primitive Selection Framework (PSF)** — where new logic lives
2. **Memory tiers** — four-tier memory separation (session / project / agent / framework)
3. **R/R/W tiering** — Read / Reason / Write agent specialization
4. **Stamps** — 3+ stamp validation gate model
5. **Bootstrap-first** — the framework is built by the framework, from the minimum self-hosting set up
6. **Dogfooding** — the framework uses its own rules on its own construction
7. **Context awareness** — never compact; dynamic hard cut; guideline budgets; session planner
8. **Plugin vs project** — the duality between what the plugin ships and what lives in the plugin's own repo
9. **Documentation as code** — size limits, taxonomy, `@include` composition
10. **Security** — public-plugin posture, opt-in telemetry, secret scanning, never leak

Load them all by reading `architecture/principles/README.md` once; each individual principle is ≤200 lines.

## Components (reference)

Populated as components are built. Phase 0 has not populated these yet — the plan specifies what will exist (see `phases/README.md`), not what exists today. Component catalogs land as:

- `components/agents.md` — agent catalog with R/R/W tier annotations
- `components/hooks.md` — hook catalog with lifecycle events
- `components/commands.md` — command catalog with phase mapping
- `components/skills.md` — skill catalog shipped to users
- `components/schemas.md` — schema catalog
- `components/rules.md` — rules catalog for the plugin's own development
- `components/mcp-servers.md` — bundled MCP server catalog

## Lifecycle (explanation)

Populated as phases materialize. Phase 0 does not populate these — the concepts exist in `principles/` and the roadmap exists in `phases/`, but the per-lifecycle-stage walkthroughs are generated by `doc-*` agents as each phase is implemented.

## Reference hierarchy

Architecture files are linked from `.claude/CLAUDE.md` (repo root) and from the directory-scoped CLAUDE.md files (`hooks/CLAUDE.md`, `agents/CLAUDE.md`, etc.) via `@include`. When you read one of those CLAUDE.md files while working in its directory, the referenced principles load automatically.
