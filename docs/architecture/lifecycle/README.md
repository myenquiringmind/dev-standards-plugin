# Lifecycle

This folder explains **how work flows through the software development lifecycle** in `dev-standards-plugin`. Each stage of the lifecycle gets its own file as it materializes.

## The nine stages + one cross-cutting

```
DISCOVER → RESEARCH → DESIGN → DEVELOP → TEST → VALIDATE → DEPLOY → OPERATE → MAINTAIN
                                                                         ↑_______↓
                                                              DOCUMENT (cross-cutting)
```

| Stage | Purpose | When | Status |
|---|---|---|---|
| Discover | Requirements elicitation, stakeholder mapping, project-state classification, R-tier scanner pipeline for brownfield work | Before any implementation | Stub (Phase 7) |
| Research | Prior-art scan, spike planning — only when the problem is genuinely novel | Gated by complexity classifier | Stub (Phase 7) |
| Design | Brainstorm alternatives, produce architecture decision, API contracts, schema design, threat model | After discover, before develop | Stub (Phase 7) |
| Develop | Scaffold → TDD → refactor. Applies stack agents (`py-*`, `fe-*`, etc.) | Main implementation | Stub (Phase 6) |
| Test | Pyramid enforcement, coverage per layer, regression generation, flake detection | Cross-cutting during develop | Stub (Phase 9) |
| Validate | Multi-language validation gate (4 CLI + subagent reviews) writing stamps | Before every commit | Stub (Phase 6) |
| Deploy | Release review, migration sequence, canary, smoke | Before merge to main | Stub (Phase 10) |
| Operate | Incident response, SLO monitoring, runbook execution | Post-deploy | Stub (Phase 10) |
| Maintain | Dependency updates, deprecation scans, flake detection | Long-term | Stub (Phase 10) |
| Document | ADR writing, runbook generation, sequence diagrams, onboarding | Cross-cutting | Stub (Phase 9) |

## Why this is empty

Phase 0 deliberately does not populate the per-stage walkthroughs. Those documents are **generated by `doc-*` agents** from the agent catalog + code + process as each phase materializes. Hand-writing them now would:

1. Duplicate content that will be generated later from the source of truth
2. Drift from the actual agents when agents evolve
3. Violate the dogfooding principle — the framework should produce its own documentation via its own agents

The principles in `docs/architecture/principles/` and the phase roadmap in `docs/phases/` are the load-bearing content for the lifecycle. Per-stage walkthroughs land when the relevant phase's doc agents run.

## When each stage's file will exist

| Stage file | Phase |
|---|---|
| `discover.md`, `research.md`, `design.md` | End of Phase 7 |
| `develop.md`, `validate.md` | End of Phase 6 |
| `test.md` | End of Phase 9 |
| `deploy.md`, `operate.md`, `maintain.md` | End of Phase 10 |
| `document.md` | End of Phase 9 |

Until then, lifecycle-specific knowledge lives in:

- `docs/phases/phase-N-*.md` — what Phase N delivers
- `docs/architecture/principles/bootstrap-first.md` — Modelling's ten interlocking mechanisms

Do not hand-write lifecycle stage docs. They are generated artifacts.
