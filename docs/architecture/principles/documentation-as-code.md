# Documentation as Code

The framework treats documentation with the same discipline as code: it has structure, size limits, tests, versioning, and mechanical enforcement. This is the fix for the monolithic-plan failure described in `@dogfooding.md` §"Worked example".

## Size limit — mechanically enforced

**Every markdown file in `docs/`, `.claude/`, and `CLAUDE.md` at any directory level is ≤200 lines.**

Enforced by `hooks/post_edit_doc_size.py` (PostToolUse on Edit/Write matching `**/*.md`). Warns at 150 lines, blocks at 200 (exit 2). Configuration at `config/doc-size-limits.json` allows per-pattern overrides.

**Exempt paths** (only):

- `docs/decision-records/v2-architecture-planning-session.md` — archived canonical plan, historical artifact
- `CHANGELOG.md` — release history
- `README.md` at repo root — user-facing project intro (first-impression file, slightly different constraints)
- `tmp/**`, `node_modules/**`, `.venv/**` — not under our control

Everything else is hard-blocked at 200 lines. No exceptions, no "just this once."

## Why 200 lines

From Claude Code's official documentation on memory: "target under 200 lines per CLAUDE.md file. Longer files consume more context and reduce adherence." This is the upper bound at which instruction-following remains reliable, per Anthropic's own best practice.

200 lines ≈ 1500-3000 tokens depending on content density. Small enough that loading 5-10 of them costs less than loading one 3000-line monolith. Big enough to hold a single architectural concept with examples.

## The Diataxis taxonomy

Documentation is split by user intent × content type:

| | Practical (doing) | Theoretical (learning) |
|---|---|---|
| **Study** (acquiring skill) | **Tutorials** (`docs/tutorials/`, deferred) | **Explanation** (`docs/architecture/principles/`, `docs/decision-records/`) |
| **Work** (applying skill) | **How-to guides** (`docs/guides/`) | **Reference** (`docs/architecture/components/`) |

Each quadrant answers a different question:

- **Tutorials** — "I'm new, teach me"
- **How-to** — "I have a goal, how do I achieve it"
- **Reference** — "I need a fact, what is X"
- **Explanation** — "I want to understand, why is it like this"

**Mixing quadrants in one document is the most common docs anti-pattern** — it confuses all four readerships simultaneously. The framework's own documentation avoids mixing.

## Composition via `@include`

Claude Code supports `@path` syntax in instruction files to include other files lazily. When a parent file is loaded and it contains `@path/to/other.md`, the referenced file loads on demand. `InstructionsLoaded` hook fires with `load_reason: include`.

This is the composition mechanism: small files (each <200 lines) compose into larger intellectual units without duplication. A CLAUDE.md at the repo root can `@include` three or four principles; those principles remain single-concern files that can also be read standalone or referenced from other CLAUDE.md files at different directory levels.

## Durability under compaction

CC's compaction algorithm preserves CLAUDE.md files fully — they're re-read from disk after compaction fires. Rules and skills are less durable (skills share a 25K post-compaction budget; rules are not explicitly preserved). **Load-bearing content that must survive compaction belongs in CLAUDE.md, not in rules or skills.**

See `@memory-tiers.md` for the full compaction behaviour.

## What goes where — the decision tree

**"I want to explain a concept"** → `docs/architecture/principles/<concept>.md`

**"I want to catalog components of type X"** → `docs/architecture/components/<type>.md` (populated by `doc-component-catalog-writer` from source of truth; hand-edits are replaced on the next regeneration)

**"I want to describe a lifecycle phase"** → `docs/architecture/lifecycle/<phase>.md` (generated by `doc-lifecycle-writer` as each phase completes)

**"I want to specify an implementation phase"** → `docs/phases/phase-<N>-<name>.md`

**"I want to record a decision"** → `docs/decision-records/adr-<NNN>-<slug>.md`

**"I want a how-to for a specific task"** → `docs/guides/<task>.md`

**"I want a tutorial for newcomers"** → `docs/tutorials/` (deferred to Phase 10+)

## Generated vs hand-written

Some documentation is hand-written (principles, ADRs, phase specs, guides, tutorials). Some is **generated** from the source of truth by `doc-*` agents:

| Category | Source of truth | Generator |
|---|---|---|
| `architecture/components/agents.md` | `agents/**/*.md` frontmatter | `doc-component-catalog-writer` |
| `architecture/components/hooks.md` | `hooks/*.py` module docstrings | `doc-component-catalog-writer` |
| `architecture/components/commands.md` | `commands/*.md` frontmatter | `doc-component-catalog-writer` |
| `architecture/components/skills.md` | `skills/*/SKILL.md` frontmatter | `doc-component-catalog-writer` |
| `architecture/components/rules.md` | `.claude/rules/*.md` | `doc-component-catalog-writer` |
| `architecture/components/schemas.md` | `schemas/**/*.json` | `doc-component-catalog-writer` |
| `architecture/lifecycle/*.md` | Phase walkthroughs from actual phase completion | `doc-lifecycle-writer` |

Hand-editing a generated file is an anti-pattern — the generator will overwrite the edits on the next run. If something is wrong with a generated file, fix the source of truth, not the output.

## Anti-patterns

- **A "comprehensive" single file.** See `@dogfooding.md` for the worked example.
- **Duplicate content across files.** If two files say the same thing, one should link to the other.
- **Stale generated files.** Regenerate via the appropriate `doc-*` agent; never hand-edit.
- **Mixed quadrants.** A "reference + tutorial + explanation + how-to" file helps nobody.
- **Files without a stated purpose.** Every file should have a top-level heading and a second-paragraph statement: "This document explains X."

## The failure case worth remembering

The original plan was a 3300-line monolith. It violated four of its own principles. The fix was decomposition + hook-enforced size limits + Diataxis taxonomy + `@include` composition. Every new document in the framework is subject to the same discipline. **When you feel tempted to write a large file, that's the signal to decompose.**
